# Domain Model

> **Version:** 1.0  
> **Status:** Accepted  
> **Last Updated:** 2026-06-29

## Purpose

This document defines the core business concepts (domain entities) of GeoQuest AI.

These concepts are independent of programming languages, frameworks, databases, and AI providers. They represent the shared vocabulary used throughout the project.

The goal of this document is to establish a common language before implementation begins.

---

## Core Domain Entities

### Player

A **Player** represents a participant in a match.

A player can:

- Join rooms
- Participate in matches
- Take turns
- Complete challenges
- Progress across the board

A player does not own game rules or match state.

---

### Room

A **Room** is a temporary multiplayer space where players gather before and during a match.

A room is responsible for:

- Managing connected players
- Tracking player readiness
- Creating a match once all players are ready

A room may exist without an active match.

---

### Match

A **Match** represents one complete play session.

A match begins when gameplay starts and ends when a winner is declared.

A match owns:

- Players
- Board
- Turn order
- Match state
- Game seed

A match cannot exist without a board.

---

### Turn

A **Turn** represents one player's opportunity to interact with the game.

A turn references:

- The active player
- Spinner result
- Challenge
- Player actions

A turn does **not** own the challenge itself.

---

### Spinner

The **Spinner** is responsible for selecting one country from the available country pool.

Its responsibilities include:

- Random country selection
- Deterministic selection using the game seed

The spinner is **not** responsible for:

- Player movement
- Scoring
- Match progression

---

### Board

The **Board** represents the playable path used during a match.

A board contains:

- Start tile
- Finish tile
- Playable tiles
- Player positions

The board does not contain geographic information.

---

### Tile

A **Tile** represents one position on the board.

A tile determines:

- Which challenge category is triggered
- Any future board-specific effects

Tiles are independent of players and gameplay flow.

---

### Country

A **Country** represents a geographic entity available during gameplay.

A country contains information required for:

- Map validation
- Gameplay
- Challenge generation

Countries are independent entities and are not owned by the spinner.

---

### Map

The **Map** represents the interactive geographic surface displayed to players.

Its responsibility is to:

- Display countries
- Validate player selections

The map displays countries but does not own them.

---

### Challenge

A **Challenge** represents a task presented to a player after a successful country selection.

Challenges determine whether gameplay continues according to the active rule set.

A challenge may be:

- AI-generated
- Database-driven
- Static
- Generated by another provider

Gameplay should remain independent of how challenges are created.

---

## Relationships

The following relationships define the current domain model.

### Room (Relationships)

- A Room hosts one active Match.
- A Room contains multiple Players.

### Match (Relationships)

- A Match contains multiple Players.
- A Match contains exactly one Board.
- A Match consists of multiple Turns.

### Board (Relationships)

- A Board contains multiple Tiles.
- A Board tracks player positions during a match.

### Turn (Relationships)

- A Turn references one Spinner result.
- A Turn references one Challenge.

### Geography

- A Map displays Countries.
- Countries exist independently of the Map.
- The Spinner selects Countries but does not own them.

---

## Design Principles

The domain model follows the following principles.

### Single Responsibility

Each domain entity owns one clearly defined responsibility.

---

### Technology Independence

Business concepts must not depend on:

- Frameworks
- Databases
- AI providers
- UI implementation details

---

### Separation of Concerns

Gameplay should remain independent from:

- Authentication
- AI providers
- Storage
- Network communication

---

### Extensibility

The architecture should allow new challenge types to be introduced without modifying gameplay logic.

Examples include:

- Multiple-choice questions
- Flag identification
- Capital city questions
- Currency questions
- Image-based challenges
- Audio-based challenges

---

## Future Evolution

The domain model is expected to evolve as GeoQuest AI grows.

Future entities may include:

- Tournament
- Team
- Achievement
- Season
- Leaderboard
- Inventory
- Badge
- Game Mode

New entities should be introduced only when they represent genuine business concepts rather than implementation details.
